# QChat 多源联网检索设计（第二阶段）

## 目标与边界

本阶段让 QChat 在第一阶段的语义决策结果为“需要联网”时，能够并行使用 DuckDuckGo 和 Bing 获取可验证的外部证据。检索结果在进入模型前会被规范化、去重、排序并截断；QChat 仍按既有人设自然回答，而非向 QQ 用户输出固定的“正在搜索”或引擎报错文案。

本设计保留第一阶段的访问和时延边界：仅主人私聊、或群聊中明确 @ 当前活跃 bot 的消息能够触发语义联网；DataAgent 不进入 QChat 的关键检索路径；普通回复不需要等待联网。显式 `/search` 与语义路由使用同一条安全证据通道。

本阶段不包含定时新闻推送、主动外发、插件自动下载安装/更新、SmartWebSearch 文本结果解析，也不添加或模拟 Tavily、百度千帆等第三方密钥。

## 架构

```text
QChat 语义路由或显式 /search
  → 多源协调器（仅启用后）
  → DuckDuckGo + Bing 并行、各自超时与熔断
  → 规范化的结构化结果
  → 规范 URL 去重 + 相似标题去重 + 确定性排序 + 数量上限
  → AgentWebResearchService 的不可信外部证据封装
  → 原有 QChat 人设回复

手工安装的 Alife-SmartWebSearch 原生插件
  → Alife 原有插件生命周期
  → 面向通用 Agent/LLM 的 XML 工具
  → 不作为 QChat 结构化证据提供者
```

多源协调器是 QChat 现有联网执行器内部的一个有界组件。它的输入是已经获得授权的搜索请求，输出是来源、标题、摘要和 URL 都可追溯的结构化搜索项；它不负责决定是否搜索、生成用户可见回复，或调用 DataAgent。

## 并行、降级与时延策略

当 `Enabled=true` 时，DuckDuckGo 和 Bing 同时启动，默认分别允许 1,500 ms。任一提供者成功时，收集截至该时刻已完成的结果，并取消尚未完成的同伴；这样快速成功不会被慢引擎拖住。一个提供者失败或在自身超时内无结果时，另一个成功提供者仍可生成证据。两个都失败时沿用第一阶段的 `research_failed` 结果，不捏造来源或内容。

调用方取消时，协调器将取消所有正在进行的提供者调用，并不发送迟到的联网反馈。每个提供者维护独立的连续失败计数：默认连续 3 次失败后打开 60 秒熔断器；熔断期间不再调用该提供者，但另一个提供者仍可工作。成功调用重置其失败计数。既有成功缓存继续存在，并在调用任何提供者前复用；缓存命中不启动联网请求。

诊断仍通过既有 QChat diagnostics 输出。QQ 用户可见的进度和回复只能由既有自然人设/模型路径形成，不能新增按引擎名称或失败原因拼接的硬编码话术。

## 结果规范化与证据约束

每个内置提供者的输出先映射到统一结构：`Source`、`Title`、`Snippet`、`Url`。缺失或无效 URL 的项不进入模型证据。URL 先去掉片段、规范化大小写及无语义的尾随分隔，再按规范 URL 精确去重；相同 URL 保留确定性优先的一项。随后对规范化标题做确定性近似比对，近似重复仅保留排序更高的一项。

保留结果按既定、稳定的打分规则排序，并最多向下游提供 `MaxMergedResults`（默认 5）项。模型仅接收这一小组结构化的来源链接和已由第一阶段实现包裹的“不可信外部上下文”；不能把搜索网页的指令、HTML 或插件生成的自然语言直接拼入系统提示词。外部上下文不得改变工具权限、访问控制或人设约束。

## 配置

新增独立的多源联网配置，默认关闭，以保持未升级实例的第一阶段行为：

| 配置项 | 默认值 | 含义 |
| --- | ---: | --- |
| `Enabled` | `false` | 启用内置多源协调器；关闭时保持现有单路径行为。 |
| `ParallelBuiltInProviders` | `true` | 启用时同时请求 DuckDuckGo 与 Bing。 |
| `PerProviderTimeoutMilliseconds` | `1500` | 每个内置提供者的独立时间预算。 |
| `MaxMergedResults` | `5` | 进入模型前的合并证据上限。 |
| `FailureThreshold` | `3` | 打开单提供者熔断器所需连续失败次数。 |
| `CircuitBreakSeconds` | `60` | 熔断保持时间。 |
| `DetectSmartWebSearchPlugin` | `true` | 仅检测手工安装插件的可用/配置状态，绝不使 QChat 依赖它。 |

配置值必须进行边界校验：超时与熔断时间为正，结果上限与失败阈值至少为 1。关闭多源功能、未安装插件、或插件缺少密钥都不改变 QChat 的现有搜索可用性。

## Alife-SmartWebSearch 集成决定

已检查远端仓库 `L134283/Alife-SmartWebSearch`。它是 Alife 原生插件，不是独立的 HTTP 服务：插件通过 `Alife.Plugin.SmartWebSearch.json` 和现有模块生命周期被发现，`SmartWebSearch.cs` 中以 `[XmlFunction]` 提供 `Search`、`SmartSummary`、`SmartChatSearch`、`HotSearch` 等能力，结果通过 `Poke(...)` 以自然语言返回。其配置需要 Tavily 或百度千帆密钥。

该插件没有为 QChat 提供诸如 `IAgentPublicSearchProvider` 的结构化证据接口。将它的 Markdown/自然语言输出反向解析为 URL 和摘要会破坏来源可信边界、排序确定性与测试性，因此明确不做。实现仅允许 Alife 的既有插件机制加载用户手工安装且已配置的包；它可向通用 Agent/LLM 暴露其 XML 工具，但不参与 QChat 的并行内置检索、结果合并、缓存、熔断或证据注入。插件不存在或没有密钥时，仅记录非用户可见诊断状态，不阻塞 QChat。

## 成功标准与测试矩阵

实现必须以可注入的提供者/时钟/取消令牌进行测试，覆盖下列行为：

1. 两个内置提供者确实并行启动，而不是串行后备。
2. 更快的成功返回不会等待较慢的提供者；未完成同伴会收到取消。
3. 一个提供者失败时仍返回另一个提供者的结果；两个失败时返回既有 `research_failed` 语义且没有伪造证据。
4. 调用方取消会传递到两个提供者，并且不产生迟到反馈。
5. 精确规范 URL 重复会去重；近似标题会以确定性规则去重。
6. 合并结果排序稳定、来源数受 `MaxMergedResults` 限制，且只有有效 URL 能进入证据。
7. 命中既有成功缓存时不调用提供者。
8. 单一提供者连续失败达到阈值后打开熔断器，并在配置窗口结束后恢复调用。
9. 未安装或未配置 SmartWebSearch 不影响 QChat 检索；新 QChat 路径不引用 DataAgent。
10. 第一阶段语义路由、显式 `/search`、私聊与群聊 @ 访问规则保持回归通过。

验证还应覆盖配置默认关闭时的兼容行为，并以完整 QChat 测试集作为回归门槛。

## 明确排除项

- 自动下载、自动升级或自动配置第三方 SmartWebSearch 插件。
- 从 `Poke(...)`、Markdown 或模型自然语言中提取结构化证据。
- 添加 Tavily/百度千帆凭据或绕过其认证要求。
- 主动新闻、定时任务或非用户提问触发的外发联网。
- 将 DataAgent 加入任何 QChat 的联网关键路径。
- 为具体 QQ 用户、群号、关键词或搜索引擎硬编码用户可见回答。
