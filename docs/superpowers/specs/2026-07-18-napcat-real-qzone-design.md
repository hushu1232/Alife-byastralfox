# NapCat 真实 QQ 空间集成设计

**日期：** 2026-07-18

**状态：** 已确认，待审阅后实施
**范围：** 夏羽与雨宫咪绪两个本地 NapCat 账号的真实 QQ 空间读取、发布、互动、图片上传和角色文案生成。

## 目标

将现有 QZone 模块从假定的 OneBot 动作名（`send_comment`、`get_qzone_comments` 等）替换为真实 QQ 空间 HTTP 运行时。NapCat 只提供当前登录账号的短期空间会话；C# 使用该会话完成实际空间请求。

本次交付包含：

1. 通过 NapCat `get_cookies` 获取 `qzone.qq.com` 会话；
2. 读取可访问的说说和评论；
3. 发布文字说说；
4. 真实评论、回复、点赞和删除当前账号自己拥有的内容；
5. 上传图片并发布图文说说；
6. 使用已配置的角色 Grok 生成候选文案，并接入既有的角色发布节奏。

DataAgent 与 LangGraph 维持分析存储职责；它们不获得 Cookie、OneBot、QZone HTTP、发布、调度或权限执行能力。

## 已确认的运行环境

- NapCat：本机 `v4.18.6`。
- 两个本机 OneBot 账号端口：`ws://127.0.0.1:3001` 和 `ws://127.0.0.1:3002`。
- NapCat 的真实动作：`get_cookies`，请求参数为 `domain: "qzone.qq.com"`，响应提供 `cookies` 与 `bkn`。
- 当前的 `OneBotQZoneRuntime` 是早期占位适配器；它不能承担真实空间操作。
- 不维护 `AllowedQZoneTargetIds` 白名单。两个角色账号没有其他好友，目标的可访问性由 QQ 空间自身关系、可见性和服务端权限决定。

## 方案选择

采用“NapCat Cookie + 直接 QZone HTTP”方案。

不使用浏览器自动化作为主路径：它会额外引入页面选择器、登录界面、验证码窗口和运行窗口依赖。也不修改 NapCat 内部模块：当前版本未提供原生空间写入动作，维护成本高且会随 NapCat 升级失效。

QQ 空间 HTTP 运行时使用已经验证的 Web 接口类别：

- 说说读取：`emotion_cgi_msglist_v6`；
- 动态读取：`feeds3_html_more`；
- 发文字/图文说说：`emotion_cgi_publish_v6`；
- 评论：`emotion_cgi_re_feeds` 或 UGC 评论接口；
- 回复：UGC 回复接口；
- 点赞：`internal_dolike_app`；
- 删除：`emotion_cgi_delete_v6` 及相应评论/回复删除接口；
- 图片上传：`cgi_upload_image`。

QQ 空间服务端若要求验证码、限制访问、拒绝权限或返回风控状态，系统如实返回该状态；不尝试绕开 QQ 的登录、验证码或权限机制。

## 架构

```text
夏羽 / 咪绪身份
      │
      ▼
对应 OneBot 连接（3001 / 3002）
      │  get_cookies(qzone.qq.com)
      ▼
NapCatQZoneSessionProvider
      │  Cookie + bkn，仅本次 HTTP 请求内存中存在
      ▼
QZoneHttpRuntime
      ├─ 读动态 / 说说 / 评论
      ├─ 发说说 / 评论 / 回复 / 点赞 / 删除
      └─ 上传图片 → 发图文说说
      │
      ▼
QZoneService + 角色格式化反馈
      ├─ 手动 QZone 命令
      └─ 既有随机调度 → Grok 候选文案 → 真实发布
```

### 会话提供器

新增 `NapCatQZoneSessionProvider`，按角色绑定已有 OneBot 连接，调用 `get_cookies`。它向 HTTP 运行时提供 Cookie、`bkn`、当前 QQ 号和请求头所需的临时会话数据。

Cookie、`bkn`、完整请求头和上传表单不进入配置、SQLite、审计、异常文本、终端输出或 Git。日志只记录操作类别、角色、成功/失败状态及 QQ 空间返回的非敏感错误码。

### 真实运行时

新增 `QZoneHttpRuntime`，作为 `IQZoneRuntime` 的真实实现。它负责表单提交、JSON/JSONP 去包装和响应映射；现有 `OneBotQZoneRuntime` 不再作为默认真实运行时。

接口模型补齐实际动作需要的元数据：说说的 `topicId` / `feedsKey` / 时间戳，评论和回复的真实 ID，以及图片上传返回的相册定位信息。删除操作只允许当前会话账号拥有的说说、评论或回复，且必须携带从读取结果获得的对象元数据。

`QZoneService` 根据真实模式创建该运行时；原来的 OneBot URL、令牌和双账号启动方式保持不变。NapCat Cookie 过期或 QQ 空间拒绝时，下一次操作重新调用 `get_cookies`，不缓存到磁盘。

### 图片来源与发布

图片上传统一接受 `QZoneImageSource`，并按以下优先级解析：

1. 主人通过 QQ 或本地管理文件交给角色的图片；
2. 项目内由对应角色生成并保留的图片输出；
3. 主人明确给出的直接 HTTP(S) 图片 URL，下载到临时内存/临时文件后上传；
4. 没有适合图片时，退回纯文字说说。

自动空间行为不会为了配图随机爬取网页、搜索图片或转载未知来源素材。每次图片上传只使用解析后的二进制数据；QZone 返回的相册标识仅用于当前图文发布和后续删除自己的图片，发布完成后不保存图片字节。临时文件位于项目运行临时目录，不提交 Git。

图片选择由角色人设驱动：夏羽与咪绪各自通过既有角色上下文选择主题与描述；媒体源的归属和角色绑定由 C# 保证，避免跨角色使用私人上传内容。

### LLM 文案与自动发布

新增窄接口 `IQZoneDraftGenerator`。其真实实现复用每个角色已配置的 Grok 客户端和完整角色记忆上下文，不新增或复制 API Key。

输入为角色身份、发布/评论场景、调度信号、简要近期空间摘要、长度限制及是否有配图；输出为一条候选文本。C# 负责限制长度、去除空内容、避免与近期已发文案重复，并将成功/拒绝/失败反馈交给已有角色格式化器。它不把系统提示、完整记忆或 Cookie 写入审计。

现有 24–42 小时随机候选、日间窗口、角色人设规则和“48 小时目标”保留。真实自动发说说由单一 `EnableQZoneAutonomyLivePosting` 配置开启：关闭时仍只生成/记录 dry-run 候选；开启时，合格候选经 Grok 生成后直接由对应角色账号发布。手动发布、评论、回复、点赞、删除和图片操作走同一真实运行时，不额外建立审批队列。

## 配置与反馈

新增的配置只包括真实所需项：

- `UseNapCatQZoneHttpRuntime`：选择真实 QZone HTTP 运行时；
- `EnableQZoneAutonomyLivePosting`：允许已启用的角色调度真实发布；
- 每个角色已存在的 OneBot URL、令牌和 QQ 号绑定；
- 图片 URL 的最大下载尺寸和上传图片数量上限，以匹配 QQ 空间接口限制。

不引入空间目标白名单、复杂审批工作流、浏览器配置或 DataAgent/LangGraph 开关。远程图片 URL 在本期只接受主人明确提供的单条直链；当将来需要让角色从预先认可的站点主动取图时，再独立增加按域名维护的网址白名单。QZone 成功、拒绝、限流和任务反馈仍使用夏羽/咪绪各自的 C# 角色格式化文本，而不是机械的通用状态句。

## 必要验证

实现后只做与真实接入直接相关的验证：

1. 以伪造 OneBot/HTTP 响应验证 Cookie 动作、请求字段和 JSONP 解析；
2. 构建目标项目和现有 QChat/QZone 测试；
3. 用户授权后，对两个真实账号分别进行一次读取、文字发布、评论或点赞、图片上传和图文发布检查；
4. 仅在真实验证成功后开启自动真实发布开关。

不会新增大规模模拟、浏览器回归套件、灰度系统或额外服务。

## 非目标

- 不绕过 QQ 登录、验证码、空间权限或风控；
- 不向 DataAgent/LangGraph 交出执行能力；
- 不增加 QQ 好友或改动现有好友关系；
- 不抓取未知来源图片作为自动配图；
- 本期不实现网址白名单；它是未来主动远程取图能力的配置项；
- 不把 Cookie、API Key、账号会话、图片二进制或本地运行状态提交到仓库。
