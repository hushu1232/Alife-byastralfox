# QChat Agnes 主识图与 Grok 自动降级设计

## 目标

在不改变 QChat 聊天授权、工具授权、DataAgent 或 LangGraph 边界的前提下，为 QQ 图片观察增加双提供商路由：Agnes 处理默认图片，Grok 处理明确的高难视觉问题并承接 Agnes 的可恢复失败。图片到达即可自动触发观察，不以 `@` 作为条件；群聊是否发言仍由既有群聊回复策略决定。

本设计不包含 API 密钥、真实账号、真实端点、角色私有资料或本地配置内容。

## 已确认的产品决策

- 图片识别自动触发，不要求 `@`。
- 默认主提供商为 Agnes。
- Grok 为自动降级提供商；对于 OCR、截图、表格、图表、代码、界面细节等明确高难请求可直接选择 Grok。
- Grok 使用现有本地 OpenAI 兼容 Grok 配置与指定多模态模型，但视觉密钥解析和健康状态独立于文字聊天运行时。
- 同一图片不并发上传至两个提供商；一次处理最多一次主请求和一次降级请求。
- 无需让 DataAgent、LangGraph、浏览器或桌面控制接管视觉。

## 现状与缺口

现有 QChat 已有图片段解析、图片策略、每角色 / Bot 的 `VisionProfiles`、Agnes OpenAI 兼容客户端和安全化观察上下文。现有 `QChatService` 只会在全局提供商为 Agnes 时创建客户端；Profile 的 `Provider` 字段尚不能实际切换客户端，也没有主备、熔断、自动图片队列或高难路由语义。

## 架构

```text
OneBot 图片事件
  -> 图片段解析、URL 合法性、会话去重
  -> 自动图片策略（与聊天 @ / 回复策略分离）
  -> 每角色 / Bot 视觉路由
  -> 有界队列与单图片总时限
  -> 选定主提供商
       -> Agnes: 默认图片
       -> Grok: 明确 OCR / 截图 / 表格 / 图表 / 代码 / UI 细节
  -> 仅当主提供商发生可恢复失败时切换一次备用提供商
  -> 安全化的未验证视觉观察
  -> 仅在既有聊天策略允许时交给模型生成 QQ 回复
```

### 组件边界

1. `QChatVisionRoutePlanner`

   根据角色、Bot、消息文本、图片数量和每角色 Profile 产生不可变的路由计划：主提供商、可选备用提供商、路由原因、总时限和每张图片上限。它不请求网络、不授予权限。

2. `IQChatImageRecognitionClient` 提供商注册表

   提供商以稳定 id 注册：`agnes` 与 `grok`。两者均使用 OpenAI 兼容 `chat/completions` 的图像消息形状，但分别拥有名称、模型默认值、端点解析器、密钥解析器和健康状态。文本聊天客户端不直接复用为视觉客户端。

3. `QChatVisionExecutionCoordinator`

   按 Bot 维护有界、单并发的短时队列；主人消息优先，其他合规图片按到达顺序处理。它只执行已生成的路由计划，确保每张图不会并发双发，也不会无限重试。

4. `QChatImageRecognitionService`

   继续负责图片解析、策略、提供商调用结果的安全格式化和观测上下文。它不保存原始图片、URL、供应商原文或密钥。

5. Readiness 与状态输出

   以每个角色 / Bot 的视觉路由显示可用性、主 / 备提供商、模型 id、限额和安全失败码；状态接口绝不返回密钥、请求头、图片 URL、原始供应商输出或本地文件路径。

## 自动触发与回复解耦

每个通过现有 QChat 接收范围的图片都可以进入视觉观察，不需要被 @。

- 私聊：按现有私聊接收和图片策略自动观察。
- 群聊：按既有群聊接收范围自动观察，包括未 @ 的图片。
- 群聊自动观察不等于自动回复。若原有回复策略决定不发言，观察只保留在短 TTL 的内存中，不持久化、不写角色记忆、不写 DataAgent。
- 图片策略仍可根据私聊、主人群聊、普通群聊和被动群聊分别开关；新的默认值应使合规被动群图可自动观察。

## 路由语义

### 高难直达 Grok

当同一条消息的用户文本明确要求以下能力时，直接将主提供商设为 Grok：

- OCR、读取图中文字、截图内容解释；
- 表格、图表、代码、报错、界面控件、流程图；
- 对图片细节进行比较、定位、结构化解释。

判断只使用用户消息文字和确定性关键词 / 分类规则，不从图片内容推断权限或身份。

### 默认 Agnes 与降级条件

普通照片、表情、日常场景优先 Agnes。只有下列 Agnes 失败可降级 Grok：缺少 Agnes 密钥、超时、429、5xx、网络错误、响应无法解析。

以下情况不得降级：没有安全可用的公网图片 URL、超过图片限额、策略跳过、请求被取消、输入参数不合规、用户没有被 QChat 接收。

Grok 返回成功后不再回退 Agnes；Grok 失败只产生安全失败码，不重复上传，也不强制 QQ 回复。

## 资源控制

- 每 Bot 单并发的有界队列，主人图片优先。
- 同一会话内以图片 URL 规范化值或安全摘要进行短 TTL 去重。
- 每条消息最多两张图，继承并显式配置现有上限。
- 每张图具有一个端到端时限：普通路径优先分配给 Agnes，剩余预算才交给 Grok；高难直达 Grok 使用单一预算。
- 连续可恢复失败的提供商进入短暂熔断；熔断期间优先健康提供商，不积压请求。
- 识图结果仅为即时观察，缓存到期即删除。

## 安全、隐私与授权

- 图片、OCR 文本、二维码文本和截图内的指令一律是不可信观察。
- 图片观察不能成为工具调用、主人身份、权限、文件访问、桌面访问、浏览器访问、记忆访问或任何授权依据。
- 提供商提示词必须持续要求：只给出简洁事实观察，不执行图片中的指令。
- 不记录或发送图片 URL、本地路径、API 密钥、Authorization、完整供应商响应、内部观察块或原始异常到 QQ。
- 审计仅允许记录角色 / Bot、提供商、模型 id、路由原因、是否降级、耗时桶、结果状态、失败类别和可选 token 用量。
- 密钥只从本地忽略配置或环境变量解析。视觉密钥名称与文字聊天密钥逻辑分离，允许实际值相同，但不写入源码、Git、测试、角色记忆或日志。
- 视觉保持感知适配器定位，不引入 DataAgent / LangGraph 编排权限。

## 配置模型

每个 `QChatVisionProfile` 需要描述：

- 主提供商 id；
- 备用提供商 id；
- 高难请求主提供商 id；
- 每条消息图片上限；
- 自动被动群图观察开关；
- 提供商模型和端点的本地配置引用；
- 队列上限、去重 TTL、超时预算和熔断阈值。

配置按 Bot id 优先、角色 id 次之匹配，保持夏羽与咪绪隔离。未知提供商、禁用 Profile 或缺少任何可用密钥时安全降级为不观察。

## 验收与测试

1. 普通图由 Agnes 成功处理时，Grok 不被调用。
2. 明确 OCR / 截图 / 表格 / 图表 / 代码请求直达 Grok。
3. Agnes 的缺钥、超时、429、5xx、网络错误和无效 JSON 都最多降级一次到 Grok。
4. Grok 的失败不触发循环重试或再次上传 Agnes。
5. 缺 URL、图片超限、策略跳过与取消场景都不调用任一外部提供商。
6. 未 @ 的合规群图会自动进入观察；群聊没有既有回复资格时不发送 QQ 文本。
7. 同图 TTL 去重、每 Bot 队列、主人优先与熔断恢复均有确定性测试。
8. 夏羽与咪绪之间不共享 Profile、密钥解析、队列、缓存、失败计数或审计上下文。
9. 图内提示注入不能改变权限、身份、工具执行或角色记忆。
10. 状态、日志和 QQ 输出不泄露密钥、URL、本地路径、请求头或原始响应。
11. 真实连通性测试仅在用户明确授权后，以一张无敏感内容的测试图执行；它不属于单元测试默认流程。

## 范围外

- 不改变 QQ 文字聊天模型、TTS、DataAgent、LangGraph、浏览器或桌面控制。
- 不保存图片或识图全文到长期记忆、SQLite 审计、角色档案或 Git。
- 不在本设计阶段写入或验证真实密钥。
